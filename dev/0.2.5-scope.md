# Monospace Design TUI 0.2.5 Scope

Date: 2026-04-18

## Purpose

Capture the scope for the next agent-integration cleanup release. The design
standard, rendering reference, pattern library, and research have advanced,
but the agent-facing layer has drifted behind them.

This is not only a documentation-sync release. It should also define the
behavioral target for a first-class `monospace-design-tui` agent: an elite
design collaborator that gives excellent guidance, communicates clearly,
asks the right questions when ambiguity matters, works comfortably with
human-in-the-loop review, and makes recommendations that are recognizably
Mono rather than generic TUI advice.

This release should bring the agent story back into alignment with:

- `monospace-tui-design-standard.md`
- `monospace-tui-rendering-reference.md`
- `monospace-tui-pattern-library.md`
- `monospace-design-tui-research.md`
- `dev/ux-research-2026-04-17.md`

## Current Assessment

The core standard is in better shape than the agent integration layer.

The current agent material is useful, but it does not yet describe or
implement an elite design agent. It reads more like defensive operating
instructions for a brittle fetcher than guidance for a strong design
collaborator.

The current problems are:

1. Agent-facing docs are stale.
2. The MCP server still exposes outdated palette data.
3. The agent workflow does not load or reference the Pattern Library by
   default.
4. The agent tier model is no longer ordered or named appropriately for the
   2026 agent ecosystem.
5. The directive is too restrictive for modern coding agents with approvals,
   sandboxing, MCP, and multi-step execution.
6. The current approach does not sufficiently specify agent quality,
   interaction style, recommendation behavior, or HITL posture.

## Concrete Misalignments

### 1. Version drift

Some agent-facing documents referred to `1.0`, while the standard line had
already moved beyond that. This release consolidates the documentation set
onto `0.2.5`.

Affected surfaces:

- `website/content/agents/_index.md`
- `website/content/agent-ref/_index.md`
- `TUI-DESIGN.template.md`

### 2. Palette drift

Agent-facing docs and the MCP server still mention `Commander` and still refer
to the old set as `8 palettes`. That is now incorrect.

Affected surfaces:

- `website/content/agents/_index.md`
- `website/content/agent-ref/_index.md`
- `TUI-DESIGN.template.md`
- `mcp-server/server.py`
- `mcp-server/README.md`

### 3. Pattern-library underuse

The recent pattern-library work introduced reusable guidance for:

- Footer Command Bar
- Focused Surface
- Master-Detail
- Expand-to-Focus
- Object-Local Actions
- Command Jump
- Selection Grammar
- Live Drill-Down

But the current agent directive and design skill do not treat the pattern
library as a first-class design layer.

Result:

- agents are grounded at the rules/reference level
- agents are not consistently reasoning at the reusable interaction-pattern
  level
- outputs risk being compliant but not pattern-coherent

### 4. Tier model is outdated

Current tier framing:

1. `CLAUDE.md` snippet
2. `TUI-DESIGN.md`
3. Claude Code Skills
4. MCP Server

This no longer reflects the real shape of current agent platforms.

As of 2026:

- Claude Code supports MCP, slash commands, hooks, and subagents
- Codex supports MCP, local approvals, multiple surfaces, and multi-agent work
- MCP is now the most portable shared integration layer across agent platforms
- vendor-native commands/skills are accelerators, not the foundational layer

### 5. Directive is too anti-agentic

The current agent directive contains rules that are too restrictive for
current agents:

- mandatory "fresh start" behavior every time
- mandatory stop after each discrete step
- very narrow fetch/process flow
- mandatory ephemeral cache cleanup at session end

These constraints were understandable as defensive scaffolding, but they now
work against the strengths of current agents.

The directive should still enforce discipline, but it should not artificially
degrade capable agent workflows.

### 6. Target agent behavior is underspecified

The current material says how to fetch and cache, but it does not define the
desired quality bar for the agent itself.

What is missing:

- how the agent should communicate with users
- when it should ask questions instead of guessing
- when it should make a recommendation instead of listing options neutrally
- how it should explain tradeoffs
- how strongly it should steer toward Mono-compliant outcomes
- how it should behave in HITL workflows

Without this, the same standard can produce a weak agent that is technically
compliant but unhelpful, overly passive, generic, or reluctant to make good
design calls.

## Recommended Tier Model for 0.2.5

Replace the current public framing with this model:

### Tier 1: Instruction Pointer

Minimal project-level instruction file:

- `AGENTS.md`
- `CLAUDE.md`
- equivalent tool-specific memory file

Purpose:

- tell the agent Mono exists
- point to the directive or MCP server
- optionally name the chosen palette

This should no longer be branded as Claude-specific.

### Tier 2: Project Manifest

`TUI-DESIGN.md`

Purpose:

- declare archetypes
- declare palette
- define overrides
- define project conventions

This remains appropriate and should stay.

### Tier 3: MCP Integration

The portable structured interface for:

- standard sections
- reference sections
- palettes
- components
- keyboard tiers
- archetypes
- design consultation

Why this moves ahead of vendor-native extensions:

- cross-agent
- structured
- machine-queryable
- best common foundation for serious adoption

### Tier 4: Agent-Native Accelerators

Examples:

- Claude slash commands / skills / hooks
- Codex skills / app workflows / automations
- Cursor or Windsurf equivalents

Purpose:

- improve ergonomics
- improve local workflow
- add platform-native automation

These should be framed as optional acceleration layers on top of the portable
system, not as the canonical deep integration tier.

## Target: Elite Mono Agent

`0.2.5` should explicitly define the intended behavior of a Mono-aligned
agent. The target is not a generic coding assistant that happens to know the
standard. The target is a design-aware collaborator with a distinct operating
style.

### Core qualities

The Mono agent should be:

- excellent at turning ambiguous requests into concrete TUI recommendations
- grounded in the Standard, Reference, Pattern Library, and Research
- opinionated enough to recommend a strong direction when one is better
- comfortable asking clarifying questions when ambiguity changes the design
- comfortable working with human review and partial approval at each stage
- able to explain tradeoffs clearly without becoming verbose or indecisive
- aesthetically literate, not just mechanically compliant

### Communication standard

The Mono agent should:

- communicate clearly and concretely
- summarize what it understood before proposing a design direction
- use concise, direct language
- state assumptions when they matter
- give recommendations first when the design evidence is strong
- offer alternatives when tradeoffs are genuinely close
- avoid generic UI advice that could fit any design system

### HITL posture

The Mono agent should treat human-in-the-loop review as a strength, not as a
burden.

Expected behavior:

- ask when archetype choice, workflow, or palette direction is materially
  ambiguous
- proceed autonomously when the next step is clear and low-risk
- surface key tradeoffs before irreversible or high-cost design moves
- present recommendations in a way that makes approval easy
- return with focused questions rather than broad open-ended uncertainty

The right model is:

- not over-asking
- not silent guessing
- not "one micro-step per approval"
- not full autonomy without checkpoints

### Recommendation behavior

The Mono agent should be comfortable saying:

- "Use this archetype"
- "Use this pattern"
- "This is the wrong palette for the task"
- "This screen is drifting away from Mono"

When asked for guidance, it should default to a recommendation, backed by
standard and pattern reasoning, rather than a shallow option dump.

### Mono alignment

The Mono agent should actively steer toward:

- aesthetic-first CUI outcomes when appropriate
- visible focus and strong state legibility
- footer discoverability
- keyboard consistency
- shallow navigation
- pattern coherence across screens
- named palette discipline
- explicit rationale tied to Mono sections and patterns

It should not be satisfied with mere rule compliance if the result is bland,
generic, or visually underdeveloped.

### Design reasoning order

The agent should be taught to reason in this order:

1. Project constraints and overrides
2. Workflow archetype
3. Screen archetype
4. Pattern selection
5. Keyboard model
6. Component selection
7. Palette and visual tone
8. Rendering details
9. Audit against standard and project conventions

This is better than the current effective behavior, which tends to reason at
the rule/reference layer too early.

## Keyboard Tier Assessment

The keyboard tiering inside the design standard remains broadly sound.

Tier 1 / 2 / 3 / 4 in the keyboard sense are not the main problem.
The larger issue is the agent-adoption tier model, not the keyboard model.

Specific judgment:

- Tier 1 global keys: still appropriate
- Tier 2 common action keys: still appropriate
- Tier 3 screen mnemonics: still appropriate
- Tier 4 does not currently exist in the keyboard model and does not need to
  be invented for this release

The keyboard guidance remains usable for current agents because agents are not
the consumer of those tiers in the same way users are. Agents need better
design-loading logic, not a rewritten key hierarchy.

## Release Goal

0.2.5 should make Mono's agent layer:

- current
- internally consistent
- platform-neutral where appropriate
- MCP-first
- pattern-aware
- compatible with modern multi-agent and approval-based workflows
- capable of producing elite design guidance rather than only procedural compliance

## Success Criteria

The release is done when:

1. No agent-facing docs mention `Commander`.
2. No agent-facing docs describe the palette set incorrectly.
3. No agent-facing docs reference the wrong standard version.
4. The MCP server returns the current palette set only.
5. The public tier model is reordered to reflect 2026 agent reality.
6. The Pattern Library is integrated into the default agent design path.
7. The directive remains disciplined but no longer artificially constrains
   capable agents into a brittle one-step-at-a-time flow.
8. The agent materials explicitly define an elite Mono-aligned behavior model.
9. The default Mono agent posture is clear, recommendation-capable, and
   comfortable with HITL collaboration.
