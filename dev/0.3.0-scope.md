# Monospace Design TUI 0.3.0 Scope

Date: 2026-04-18

## Purpose

`0.3.0` should make Monospace Design TUI capable of supporting **workflow
design, ASCII wireframe generation, workflow and screen spec generation, and
interactive TUI-first prototyping**.

This release should make Mono useful earlier in the design process, not only
later at the standards/compliance stage.

## Core Goal

The goal for `0.3.0` is:

- use Claude / Codex / similar agents as the primary interface
- use Mono-authored tools to inspect product intent and existing project
  structure
- propose wireframe-level artifacts for:
  - overall navigation pattern
  - workflows
  - screens inside workflows
- support HITL adjustment of workflow wireframes and screen wireframes
- generate workflow and screen specs as explicit artifacts

The important constraint is:

- workflows and screens are represented as **ASCII/TUI artifacts**, not web
  mockups

## Architecture Direction

The initial technical direction for `0.3.0` is:

- **Python-first** implementation
- canonical design artifacts stored as **YAML specs**
- primary artifact types:
  - navigation spec
  - workflow spec
  - screen spec
- **ASCII** as the primary review/output form at this stage
- ASCII produced via a **projection function** from the canonical YAML specs
- a **minimal semantic markup layer** only where needed to preserve structure

The key decision is:

- ASCII is the primary human-facing output for `0.3.0`
- YAML is the canonical machine-facing representation
- minimal semantic markup exists only to preserve:
  - regions
  - components
  - actions
  - focus
  - state

This avoids over-designing a full DSL too early while still preserving enough
semantic structure for future richer renderers and tooling.

## Problem Statement

Today, Mono is strong at:

- prescriptive design rules
- rendering rules
- interaction patterns
- agent guidance for compliant implementation

But it is not yet strong at:

- turning product intent into workflow structure
- turning workflow structure into screen inventories
- turning screen inventories into low-fidelity ASCII wireframes
- using those wireframes as reviewable, revisable design artifacts
- steering prototypes toward TUI instead of web

`0.3.0` should close that gap.

## Canonical Artifact Model

The canonical Mono artifacts for `0.3.0` should be:

### 1. Navigation Spec

Represents:

- top-level sections or workspaces
- global routing structure
- major cross-workspace paths
- navigation rules

Format:

- YAML

### 2. Workflow Spec

Represents:

- workflow purpose
- stages or steps
- transitions
- HITL checkpoints
- assumptions
- linked screens

Format:

- YAML

### 3. Screen Spec

Represents:

- screen identity
- purpose
- regions
- components
- primary actions
- keyboard/focus model
- transitions
- linked workflow
- linked patterns

Format:

- YAML

### 4. ASCII Projection

Represents:

- human-reviewable output derived from the canonical specs

Expected forms:

- navigation overview
- workflow outline
- screen wireframe

This should be generated from the canonical spec rather than being treated as
the authoritative source of truth.

### 5. Minimal Semantic Markup

This is not intended to be a full Mono DSL in `0.3.0`.

It should exist only where needed to preserve semantic structure for
projection, review, and future tooling.

Minimum preserved semantic concepts:

- regions
- components
- actions
- focus
- state

## In Scope

## 1. Agent-First Design Flow

Agents should be the primary user interface to the system.

Expected model:

- user describes product intent or points at a codebase
- agent uses Mono-authored tools and docs
- agent proposes:
  - navigation model
  - workflows
  - screen set
  - wireframes
  - specs
- user reviews and revises through HITL checkpoints

This release should define that flow clearly in docs, prompts, and tool
contracts.

This release should also introduce a **second agent/skill track** that is
separate from the current implementation-oriented Mono agent material.

The existing agent material is centered on:

- standards retrieval
- implementation alignment
- compliance and audit

`0.3.0` needs a parallel track centered on:

- workflow design
- screen generation
- wireframe generation
- spec generation
- HITL revision of design artifacts

This new design-generation agent/skill set must:

- succeed at its own workflow/spec/wireframe generation work
- start from Mono guidance by default
- actively avoid drift from the Monospace Design TUI system
- allow exceptions only when explicitly required by the user or the project

It should not be a generic wireframe generator.
It should be a Mono-native design-generation agent.

## 2. Overall Navigation Pattern Generation

Mono should be able to generate an application-level navigation pattern.

Expected outputs:

- stable root navigation model
- top-level workspaces or sections
- cross-workspace movement rules
- routing heuristics
- entry/exit paths

Output form:

- navigation spec (YAML)
- optional navigation ASCII overview

## 3. Workflow Generation

Mono should be able to propose workflow-level artifacts from product intent.

Expected outputs:

- workflow name and purpose
- entry conditions
- exit conditions
- ordered stages or steps
- transitions between stages
- required HITL checkpoints
- onward transitions into other workspaces or workflows

Output form:

- workflow spec (YAML)
- workflow ASCII outline

## 4. Screen Generation Inside Workflows

For each workflow, Mono should be able to propose the screen set.

Expected outputs:

- screen inventory
- screen IDs
- screen purposes
- primary actions
- transitions between screens
- archetype mapping
- pattern assignments

Output form:

- screen spec set (YAML)
- screen relationships tied back to workflows

## 5. ASCII Wireframe Generation

Mono should generate low-fidelity TUI wireframes as first-class artifacts.

Expected properties:

- ASCII-first
- generated from canonical YAML specs
- structural rather than polished
- consistent format
- annotated with key information
- suitable for review and iteration

Expected artifact types:

- overall shell / app map wireframe
- workflow wireframes
- screen wireframes

## 6. HITL Revision Flow

Mono should support revision of workflows and screens with a human in the loop.

This does not need full artifact-graph infrastructure yet, but it does need a
clear review loop.

Expected capabilities:

- present recommendation
- ask focused clarification questions when needed
- revise workflow wireframes
- revise screen wireframes
- preserve screen IDs and workflow structure where possible
- keep outputs readable and reviewable

## 7. Workflow and Screen Specs

Mono should generate explicit specs, not only wireframes.

These specs should be canonical YAML artifacts for `0.3.0`.

Expected workflow-spec fields:

- workflow ID or name
- purpose
- entry conditions
- exit conditions
- stages
- transitions
- HITL checkpoints
- assumptions

Expected screen-spec fields:

- screen ID
- purpose
- entry conditions
- primary actions
- keyboard/focus model
- regions and components
- transitions
- linked workflow
- linked patterns

## 8. TUI-First Prototyping Guidance

`0.3.0` should explicitly steer prototypes toward TUI form when the target
product is terminal-native.

Expected additions:

- guidance for replacing web prototypes with TUI prototypes
- definition of prototype levels
- guidance for navigable Textual-first prototypes
- distinction between:
  - wireframe
  - prototype
  - implementation-ready spec

## 9. Margie Pilot

Margie should be used as the concrete pilot for `0.3.0`.

Target inputs:

- navigation notes
- workflow notes
- low-fidelity wireframe notes
- the existing `RQ-02` web prototype

Target outputs:

- Mono navigation spec
- Mono workflow specs
- Mono screen specs
- ASCII wireframes
- TUI-first prototype direction replacing the existing web prototype

## Primary Deliverables

### Documentation

New or expanded docs should define:

- canonical YAML spec format for:
  - navigation
  - workflow
  - screen
- workflow generation format
- wireframe grammar and annotations
- workflow spec format
- screen spec format
- ASCII projection rules
- minimal semantic markup rules
- interactive TUI prototyping guidance
- HITL review loop for workflow and screen revision

### Agent Guidance

Update the agent layer so it can:

- generate navigation patterns
- generate workflows
- generate screen sets
- generate ASCII wireframes
- generate workflow and screen specs
- recommend TUI-first prototypes instead of web mockups

This should be implemented as a second design-generation skill/agent track,
not as a small extension of the current implementation/compliance guidance.

### Mono-Authored Tooling

Mono-authored tools should support:

- reading product intent
- reading enough project context to propose screens and workflows
- producing structured YAML navigation/workflow/screen artifacts
- projecting those artifacts into ASCII outputs
- supporting review and revision at the artifact level

This does not require full codebase inference in `0.3.0`, but it does require
tools that are explicitly designed for wireframe and workflow generation.

## What Makes 0.3.0 Valuable

For `0.3.0` to be genuinely useful, it must do more than emit one-off ASCII
screens.

It should let a developer or designer move through this chain:

1. describe product/workflow intent
2. get a proposed navigation model
3. get proposed workflows
4. get proposed screens
5. get ASCII wireframes
6. revise with HITL
7. get structured workflow and screen specs
8. use those specs to guide implementation

If it does that well, `0.3.0` becomes a meaningful design toolchain rather
than just another documentation release.

## Out of Scope for 0.3.0

These are important, but should remain out of scope for this release unless
capacity expands significantly:

- deep codebase understanding:
  - current routes / screens / flows
  - current TUI framework structure
  - current domain objects and actions
  - existing keyboard model
  - what is already implemented vs proposed
- full current-state inference and migration support
- artifact graph infrastructure
- full round-trip diff and merge for artifacts
- formal cross-artifact consistency engine
- detailed implementation handoff schema
- sophisticated prototype-state lifecycle management
- automatic traceability of every recommendation to codebase evidence
- cross-framework structural inference across Textual / Ratatui / Bubble Tea / Ink

These are recorded in `dev/roadmap.md` under `## Unplanned`.

## Design Principles

- **TUI first**: if the product is terminal-native, design artifacts should
  default to TUI form.
- **ASCII first**: early design outputs should be lightweight, legible, and
  editable.
- **Canonical spec first**: YAML is the source of truth; ASCII is a projection.
- **Structured outputs**: wireframes alone are not enough; specs matter.
- **HITL by default**: workflow and screen design should include review points.
- **Mono alignment**: generated artifacts must use Mono archetypes, patterns,
  and keyboard conventions.
- **Recommendation-forward**: the system should recommend a direction, not only
  list options.

## Success Criteria

`0.3.0` is successful if:

1. A user can use Claude / Codex / similar agents as the primary design interface.
2. Mono-authored tools can produce:
   - overall navigation pattern
   - workflows
   - screens within workflows
   - canonical YAML specs for each of those artifacts
3. Mono can represent workflows and screens as ASCII artifacts.
4. Mono can revise workflow and screen wireframes with HITL.
5. Mono can generate workflow specs and screen specs.
6. ASCII output is produced from a projection function over canonical specs.
7. A new Mono-native design-generation skill/agent track exists and stays
   aligned with the design system unless the user explicitly overrides it.
8. The recommended prototype form for terminal-native products is TUI, not web.
9. Margie can be used as a real pilot for this design flow.

## Release Question

The key question for `0.3.0` is not:

- "Can Mono describe a TUI?"

The key question is:

- "Can Mono help design a TUI system from workflow intent to reviewable
  wireframes and specs, using TUI-native artifacts all the way through?"
