# Monospace Design TUI 0.3.0 Roadmap

Date: 2026-04-18

## Release Theme

`0.3.0` should make Monospace Design TUI usable for **generating wireframes,
workflow specs, and interactive TUI design artifacts**, not only for writing
static standards and references.

The goal is to move Mono from:

- a design specification
- a rendering reference
- a pattern library

to also being:

- a workflow design language
- a wireframing system
- a TUI-first prototyping method
- an agent-friendly design generation system

## Why 0.3.0

The current Mono corpus is good at answering:

- what a compliant TUI should look like
- which interaction patterns and keyboard conventions should be used
- how to render borders, shadows, colors, and components

But it is still weak at the earlier design stages:

- mapping task flow into screen flow
- generating low-fidelity TUI wireframes quickly
- iterating on alternative workflow designs
- turning product requirements into interactive terminal prototypes

This is the gap `0.3.0` should close.

## Product Direction

Mono should support a design workflow like this:

1. capture product/workflow intent
2. identify workflow archetype(s)
3. identify screen archetype(s)
4. assign reusable interaction patterns
5. generate low-fidelity ASCII wireframes
6. generate navigable TUI prototypes
7. review with HITL checkpoints
8. refine toward implementation

This should become a first-class Mono capability, not an ad hoc downstream use
of the Standard.

## Core Outcome

After `0.3.0`, a team should be able to hand Mono:

- a workflow description
- a product concept
- a set of user tasks
- a requirements note

and get back:

- workflow outlines
- screen inventories
- ASCII wireframes
- keyboard maps
- pattern assignments
- low-fidelity interactive TUI prototypes

## Major Workstreams

## 1. Workflow Design Layer

Mono already has workflow archetypes, but they need to become more operational
as design-generation inputs.

### Goals

- make workflow archetypes generation-ready
- define screen-sequence templates for each workflow
- define transition rules between workspaces and screens
- define where HITL checkpoints belong in each workflow type

### Needed additions

- workflow specification format
- workflow-to-screen mapping guidance
- explicit entry/exit conditions
- transition vocabulary:
  - push
  - pop
  - tab switch
  - modal
  - stage advance
  - cross-workspace jump
- required checkpoint vocabulary:
  - approve
  - revise
  - continue
  - cancel
  - collect more
  - publish

### Deliverables

- new workflow-generation guidance section
- machine-readable workflow schema or template
- examples of generated workflows from product briefs

## 2. Wireframe System

Mono should define a standard wireframe format for TUI-first design.

### Goals

- generate consistent low-fidelity ASCII wireframes
- make wireframes auditable and comparable
- keep them structural, not prematurely visual

### Needed additions

- canonical wireframe grammar
- standard annotation format
- screen IDs and naming conventions
- required wireframe metadata:
  - screen ID
  - purpose
  - entry conditions
  - primary actions
  - keyboard focus model
  - regions and components
  - onward transitions

### Deliverables

- a new wireframe reference doc
- a wireframe template library
- examples for each major archetype
- examples for workflow-specific screens such as:
  - request framing
  - approval review
  - progress monitor
  - evidence detail
  - report composition

## 3. Interactive Prototype Guidance

Mono should support **interactive TUI prototyping** explicitly.

### Goals

- define what counts as a prototype vs implementation
- define the minimum fidelity for a useful TUI prototype
- make it easy to build clickable/navigable terminal mockups

### Prototype levels

- Level 0: static ASCII wireframe
- Level 1: navigable shell with fake state transitions
- Level 2: interactive data-entry prototype with local mock state
- Level 3: workflow prototype with simulated job progress, approvals, and outcomes

### Needed additions

- prototype-state guidance
- mock-data guidance
- transition simulation guidance
- “do not overbuild” guardrails
- distinction between:
  - prototype-only controls
  - implementation-ready controls

### Deliverables

- a new interactive prototyping guide
- Textual-first prototype patterns
- prototype shell examples
- guidance for turning a web prototype concept into a TUI prototype instead

## 4. Agent Generation Support

The Mono agent layer should be able to generate workflows and wireframes
deliberately, not just infer screens ad hoc.

### Goals

- make workflow + wireframe generation a first-class agent task
- make agents produce TUI-first prototypes instead of reaching for web UIs
- support structured design outputs before code generation

### Needed additions

- a dedicated wireframe-generation mode
- a dedicated workflow-design mode
- guidance for recommending TUI prototypes over web mockups when the target
  product is terminal-native
- output contracts for:
  - workflow spec
  - screen inventory
  - wireframe set
  - prototype plan

### Deliverables

- updated agent directive
- updated `mono-tui-design` skill
- MCP consultation prompt additions for:
  - workflow design
  - wireframe generation
  - interactive prototype planning

## Design Principles for 0.3.0

- **TUI first**: when the product is terminal-native, prototypes should be in
  TUI form by default, not web.
- **Structure before polish**: wireframes and workflow specs come before final
  visual design.
- **Interactive when useful**: prototype interactivity should clarify state
  and flow, not imitate full backend behavior.
- **HITL by design**: workflows should encode approval and review points
  explicitly.
- **Mono alignment**: generated outputs must use Mono archetypes, patterns,
  keyboard conventions, and palette discipline.

## Non-Goals

`0.3.0` should not try to:

- become a full visual design tool
- replace implementation frameworks
- generate production-ready TUI apps from prose alone
- define every possible product workflow exhaustively

The goal is better design generation and better TUI-first prototyping, not
magic scaffolding.

## Release Success Criteria

`0.3.0` is successful if:

1. Mono can generate a structured workflow outline from a product task.
2. Mono can generate consistent low-fidelity ASCII wireframes.
3. Mono has explicit guidance for interactive TUI prototyping.
4. The agent layer can recommend TUI prototypes instead of defaulting to web.
5. Margie can use Mono to replace its current web prototype with a TUI-first
   prototype flow.
6. Designers and developers can move from concept -> workflow -> wireframe ->
   interactive prototype with one coherent Mono process.
