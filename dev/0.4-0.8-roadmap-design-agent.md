# Monospace Design TUI 0.4-0.8 Roadmap: Design Agent

Date: 2026-04-18

## Purpose

This roadmap describes a pragmatic path from the expected post-`0.3.0` Mono
state to a `0.8` release where Mono has:

- a real Mono DSL
- a highly effective design-generation agent and skill pair
- a strong MCP surface
- compatibility with Claude Code, Codex, and other agentic coding tools
- clear and usable tool contracts
- the ability to generate:
  - ASCII
  - TUI-oriented prototype/runtime outputs
  - framework-targeted outputs such as Textual and Bubble Tea

This roadmap assumes `0.3.0` already delivers:

- Python-first tooling
- canonical YAML navigation / workflow / screen specs
- ASCII as projected primary output
- basic design-generation agents/skills
- TUI-first workflow and wireframe generation

## Why 0.3.0 Is Still The Right Step

Before Mono can succeed at the `0.4` through `0.8` agenda, it needs to prove
the artifact loop introduced in `0.3.0`.

`0.3.0` is the right next step because it establishes the substrate that later
releases depend on:

- agent-first design flow
- canonical navigation / workflow / screen artifacts
- ASCII wireframe projection
- workflow and screen specs
- HITL revision of reviewable TUI-native artifacts

This matters because the later roadmap assumes a real semantic layer, not an
imagined one. A good DSL should emerge from repeated use of stable artifacts,
not from premature syntax design.

If Mono skipped ahead too early to the `0.5+` work, it would risk:

- inventing a DSL before the core artifact model is proven
- overfitting to hypothetical use cases
- building runtime/projection machinery before the semantics are stable
- coupling too much architecture to one generation of agents or one framework

What `0.3.0` provides to users, even before the DSL exists, is already
significant:

- a way to use Claude / Codex / similar agents as the front door to TUI design
- a way to turn product intent into navigation models, workflows, screens,
  specs, and ASCII wireframes
- a way to review and revise those artifacts through HITL
- a way to prototype terminal-native products in TUI form rather than defaulting
  to web mockups
- a way to move from concept to implementation with more structure and less
  ambiguity

So the logic of this roadmap is:

1. `0.3.0` proves the design artifact model
2. `0.4` hardens the artifact loop
3. `0.5` formalizes stable recurring structure into a DSL
4. `0.6` uses that DSL for projection
5. `0.7` hardens MCP and cross-agent compatibility
6. `0.8` reaches the first mature platform release

`0.3.0` is not the end-state. It is the necessary first platform step.

## End State: 0.8

By `0.8`, Mono should feel like a real design system runtime for terminal
applications, not only a standard plus helper prompts.

Expected `0.8` capabilities:

- define a Mono DSL for:
  - regions
  - components
  - focus targets
  - transitions
  - footer keys
  - inspector panes
- generate ASCII from the DSL
- generate prototype/runtime views from the DSL
- provide a strong agent + skill pair for design generation and revision
- provide a strong MCP surface with stable tool contracts
- support Claude Code, Codex, and similar agentic coding environments
- make tool use clear enough that the human can understand:
  - what was inferred
  - what was generated
  - what is proposed
  - what requires approval

## Guiding Strategy

The right path is incremental:

1. get the canonical artifact model working
2. strengthen the agent/tool loop around those artifacts
3. introduce a minimal semantic DSL
4. turn the DSL into a projection source for ASCII and runtime views
5. harden MCP and cross-agent compatibility
6. add framework-specific output targets

Do not try to jump straight from YAML specs to a full compiler.

## Core Principles

- **Spec before DSL**: the DSL should emerge from repeated spec patterns, not from speculation.
- **ASCII remains important**: ASCII should remain a first-class review format even after the DSL exists.
- **Agents are front ends, not the source of truth**: the source of truth is the artifact/spec/DSL layer.
- **MCP is the backbone**: vendor-native skills and tools sit on top of the portable protocol layer.
- **Mono alignment is non-optional by default**: agents and tools should follow Mono unless the user explicitly overrides it.
- **Projection over duplication**: ASCII, Textual, Bubble Tea, and other outputs should be projected from a common semantic source.

## 0.4 — Harden the 0.3 Artifact Loop

### Theme

Make the YAML spec pipeline reliable enough that real teams can use it
repeatedly without everything collapsing into one-off documents.

### Primary goals

- stabilize the canonical YAML specs:
  - navigation
  - workflow
  - screen
- define projection contracts from spec -> ASCII
- improve review/revision flow for generated artifacts
- strengthen the first design-generation agent/skill pair

### Key deliverables

- formal YAML schema definitions
- validation tooling
- deterministic ASCII projection rules
- revision operations such as:
  - add screen
  - remove screen
  - split stage
  - merge stage
  - add inspector pane
  - move action to footer
- first pass artifact lifecycle:
  - draft
  - reviewed
  - approved

### Agent/skill work

- separate design-generation skill/agent track fully from implementation/audit
- improve HITL checkpoints
- improve recommendation clarity
- improve artifact naming and traceability

### Success

Mono can reliably generate and revise YAML specs and ASCII wireframes for a
real project such as Margie.

## 0.5 — Introduce the Minimal Mono DSL

### Theme

Move from “specs with implied structure” to an explicit semantic design
language, but keep it intentionally small.

### Primary goals

- define the first Mono DSL
- keep the DSL tightly scoped to screen and workflow semantics
- map YAML artifacts into the DSL cleanly

### DSL scope

The initial Mono DSL should cover:

- regions
- components
- focus targets
- transitions
- footer keys
- inspector panes

Optional early additions if needed:

- state variants
- modal overlays
- list/detail relationships

### Deliverables

- `mono-dsl.md` or equivalent reference
- parser/validator in Python
- YAML -> DSL normalization step
- DSL -> ASCII projection

### What not to do

- do not build a giant syntax surface
- do not try to capture every framework nuance
- do not optimize for hand-authoring first; optimize for clarity and projection

### Success

The Mono DSL is expressive enough to represent the majority of generated
workflow and screen designs from `0.4`.

## 0.6 — Runtime and Prototype Projection

### Theme

Turn the DSL into something that can project not only to ASCII, but also to
interactive prototype/runtime views.

### Primary goals

- generate ASCII from DSL
- generate prototype/runtime views from DSL
- define the boundary between prototype projections and implementation targets

### Projection targets

- ASCII review output
- prototype shell output
- runtime-like preview output

At this stage, the main target should likely be:

- Textual-first preview/prototype projection

### Deliverables

- DSL -> ASCII renderer
- DSL -> prototype projection interface
- initial Textual prototype/runtime generator
- state simulation model for design-time previews

### Important distinction

There should be a clear separation between:

- review projection
- prototype projection
- implementation handoff

### Success

A Mono design can be represented once semantically and then projected into:

- ASCII
- an interactive TUI prototype

without redefining the screen manually.

## 0.7 — Strong MCP and Cross-Agent Tooling

### Theme

Make Mono’s design system tooling highly usable from multiple coding harnesses.

### Primary goals

- strengthen MCP around the DSL/spec layer
- make tool interfaces clearer and more stable
- ensure cross-agent compatibility

### MCP expectations

By `0.7`, MCP should expose stable design-generation tools such as:

- generate navigation spec
- generate workflow spec
- generate screen spec
- revise artifact
- render ASCII
- render prototype projection
- validate spec / DSL
- explain recommendation provenance

### Compatibility goals

- Claude Code
- Codex
- other MCP-capable coding harnesses

### Tool-use clarity goals

The user should be able to see:

- what input the tool consumed
- what assumptions were made
- what artifact was generated
- what changed on revision
- what still needs HITL approval

### Deliverables

- MCP redesign around artifact/DSL operations
- stronger MCP README and tool contract docs
- compatibility examples for Claude Code and Codex
- stronger prompts for agent coordination with Mono tools

### Success

Mono feels like a coherent design system service that multiple agent clients
can use predictably.

## 0.8 — Mature Design Agent Platform

### Theme

Reach the first “serious platform” version of Mono as a design-generation
system.

### Primary goals

- mature Mono DSL
- highly effective design-generation agent and skill pair
- strong MCP surface
- multi-target generation
- strong cross-agent compatibility

### Agent expectations

By `0.8`, the main design-generation agent/skill pair should:

- understand Mono deeply
- generate and revise navigation/workflow/screen artifacts
- use the DSL and projections correctly
- maintain Mono alignment by default
- explain recommendations clearly
- support HITL revision naturally
- know when to ask and when to proceed

### DSL expectations

By `0.8`, the Mono DSL should be stable enough to represent:

- regions
- components
- focus targets
- transitions
- footer keys
- inspector panes
- the most common state and layout relationships needed for TUI-first design

### Output expectations

By `0.8`, Mono should support generation of:

- ASCII
- Textual-oriented prototype/runtime views
- Bubble Tea-oriented outputs
- possibly additional targets if the abstraction remains clean

### Deliverables

- stable Mono DSL reference
- stable Python implementation
- strong design-generation skill/agent pair
- strong MCP service
- framework-target projection docs and examples
- end-to-end examples from intent -> DSL -> ASCII -> prototype/runtime output

### Success

At `0.8`, Mono should feel like:

- a design system
- a semantic design language
- an agent-usable toolchain
- a TUI design/prototyping platform

## Cross-Cutting Efforts

These will likely span multiple releases from `0.4` through `0.8`.

### 1. Artifact traceability

Every major proposal should be able to cite:

- relevant Mono archetype
- relevant pattern(s)
- relevant codebase evidence when available
- workflow assumptions

### 2. Revision safety

Mono should preserve stable IDs and relationships where possible during
revisions.

### 3. Spec/DSL clarity

The human should not need to reverse-engineer what the tool generated.

### 4. Projection consistency

Different outputs should remain recognizably derived from the same semantic
source.

### 5. Multi-framework discipline

Support more targets only when the shared abstraction remains clean.

## Likely Tech Stack Path

### 0.4-0.6

- Python core
- YAML specs
- Python schema/validation
- Python DSL parser/validator
- Python ASCII renderer
- Python prototype projection layer

### 0.7

- Python MCP server hardened around artifact/DSL operations
- stronger cross-agent tooling

### 0.8 and beyond

- keep Python as the primary implementation if it remains productive
- consider a Rust core only if:
  - validation
  - parsing
  - multi-target projection
  - or runtime determinism
  become limiting factors

In general:

- Python is the pragmatic early path
- a stricter core can come later if justified

## Risks

- inventing too much DSL too early
- building framework-specific features into the semantic core
- letting agent prompts outrun the artifact model
- coupling too tightly to one coding harness
- losing ASCII readability while chasing richer runtime outputs

## Practical Question for Each Release

### 0.4

Can Mono make YAML + ASCII generation reliable?

### 0.5

Can Mono define a minimal DSL without overreaching?

### 0.6

Can the DSL project into both ASCII and interactive prototype views?

### 0.7

Can multiple agent harnesses use Mono tools clearly and predictably?

### 0.8

Can Mono act as a real design-generation platform for terminal-native products?
